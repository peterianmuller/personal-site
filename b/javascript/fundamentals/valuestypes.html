Values & Types AKA How Javascript Stores Information

Computers store information as data. Programs use data, and this data is contained within values. These values are stored to the computer's hardware memory at runtime.

Values have different types. There are primitive data types:

strings

numbers

booleans

undefined

null

There are also complex value types:

plain objects

arrays

functions

Note that below, the data that appears after the comments are not visible to the programmer as is. They only exist in the computer's hardware

If you ever want to see what type something is you can use the handy typeof() method. Note that the typeof operator will return a string with the type

typeof("cat"); // "string"
typeof(6); // "number"
typeof(true); // "boolean"
typeof(undefined); // "undefined"
typeof(null); // "object"

 More info on why typeof(null); // object

One way we create values is to type in the desired data into our program. This typing of the data is creating a "literal". When an interpreter finds our literal, a value is created in hardware memory which has a type and contains the data that we've written into the program.

Whenever we type in a value, the interpreter saves it to a place in the computer memory with the type and the data contained by the value. Below are some examples of literals.

String literal:

"Steph Curry"

Type: String. Data: "Steph Curry"

300

Type: number. Data: 300

Type: Boolean. Data: false.

Booleans are used to represent two-sided ideas: on vs off, weak vs. strong signal. The tradition of having opposing ideas such as these is central to computing and most languages have a stat type to model opposition. in this case using the words true and false. Note that with Booleans, there are no quotation marks around the data, yet the interpreter will not parse the data as a variable name. This is because true and false are special forms that the interpreter understands as Booleans and will treat them as such upon seeing them in code.

A note about undefined and null.

null;

type: Null, data: null

undefined;

type: undefined, data: undefined

null is a literal that represents an empty value and is normally set on purpose to reflect nothingness/non-existance.

Undefined is a value that is returned when no data has been set to that value. However the interpreter can return undefined when trying to access an unset property of a declared variable/object. For example:

var peter =;
console.log(peter); // undefined

A variable that is declared without pointing to any value carries the value undefined. Also, a function with no return statement will return the default value of undefined.

However, like we've seen we can set a value to null on purpose to represent the idea of nothingness. Below we have a variable that we are actively setting to nothingness.

var willTheWarriorsLoseTheFinals =null;
console.log(willTheWarriorsLoseTheFinals); // null;

Also note that the above values are saved to a computer's memory when a program is running. Once the program is done running, the values dissipate back into bits. These bits will later be used to form other values.

So that's dope that Javascript saves all data as values with specific types. But what's the big deal about types? What have types done for me lately?

One dope thing about types is integral methods that aid the type of work programers will do in reasoning about them. For example, the type Number includes with it math operators that aid the execution of mathmatical formulas.

Let's say we want to do some math formulas. If we're using type Number, we get access to ready-made operators that perform basic math functions we all loved growing up with: +, -, %, *

3 + 4; //7
3 * 4; //12
3 % 4; //1 WTF?

% is our homie "modulo", or "remainder".

x % y returns the remainder of dividing x by y.

7 % 5; //2. This is because 7/5 = 1 remainder 2. The remainder 2 is what we get back from the homie modulo.

Note that the + operator also performs differently if the data type is of type String and type Number.

Another important thing about types is that the interpreter interprets types based on different rules.

For example, let's think about /. If we use this symbol in a string, than the interpreter understands that it's part of the string. However if we use / in context of number literals than this symbol becomes a division operator.

"10/5"; //"10/5"

10 / 5; //2

Thus types provide us with different operators that reason about values in specific ways according to the type and also help guide the interpreter in it's evaluation of the data at hand.
