JavaScript is a liberal language that allows the programmer to pass in values that could lead to undesired outcomes. For example, the JS interpreter has a tendency to convert values to other types based on what it thinks the program should read.

Haverbeke has an excellent explanation of this on page 18.


console.log(5 + null);
// 5

console.log('5' - 1);
// 4

console.log('5' + 1);
// 51

The first example changes the value null to 0 and then completes the addition operation. However, null is not typically meant to be added to another value as if it were an integer.

Next we see that the string '5' has been converted to a number and the result is 4, which is the correct result of the operation had we passed in 5 + 1.

The final example here has the same operands, but we've changed the operator to +. There is a + for strings which will result in string concatenation - a process very different from addition of numbers.

Some more examples


console.log('five' + null);
// fivenull

console.log('five' - null);
// NaN

Here the first example results in a stringification of null so that the interpreter can complete a string concatenation of 'five' and 'null', resulting in 'fivenull'. The second example contains the same operands yet results in NaN, which makes sense. We can't reason about strings using the - operator, so the calculation 'five' - null doesn't really make sense.

The conclusion that Haverbeke makes for type coercion is that when we pass a value to an operator that we don't mean to use, such as '5' + 1 if we want to do addition and not string concatenation, JS will "quietly convert that value to the type it wants"(18).

Hence, we should be careful when passing different value types into functions and operations because we won't always be able to predict what JavaScript will decide our values should be converted to.
